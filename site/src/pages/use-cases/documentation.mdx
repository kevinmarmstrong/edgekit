Documentation is the strongest use case for edgekit. The content is structured, the domain is bounded, and the user's intent is clear: they need an answer to a specific question. A small model with good retrieval handles this better than a search bar and often better than reading the docs manually.

## The problem

Documentation sites share a common failure mode: users cannot find what they need. The information exists, but it is scattered across pages, buried in subsections, or described with terminology the user does not know yet.

Traditional search (keyword matching) fails when the user's vocabulary does not match the documentation's vocabulary. A user searching for "how to change colors" will miss a page titled "Theming Configuration" unless the keyword appears in the body text.

AI search solves this by understanding intent. The query "how do I change the primary color" matches content about theme tokens, CSS custom properties, and configuration files -- even if the word "color" does not appear in the section heading.

## The solution

Edgekit adds an AI assistant to your documentation site that:

1. Understands natural-language questions about your docs
2. Retrieves the most relevant sections using vector similarity search
3. Generates a concise answer grounded in your actual documentation
4. Cites the specific pages where the answer was found
5. Runs entirely in the visitor's browser -- no API key, no server, no cost per query

Because the model runs locally, you can deploy this to a documentation site that gets 10 or 10 million queries and pay nothing for inference.

## Implementation

### Step 1: Index your documentation

Edgekit's CLI reads your markdown or HTML files and builds a static content index with pre-computed embeddings.

```bash
npx @edgekit/cli build \
  --input ./docs \
  --output ./public/content-index.json
```

The CLI:
- Splits your content into semantic chunks (respecting heading boundaries)
- Generates embedding vectors for each chunk
- Produces a single JSON file (~1-5 MB depending on doc size)
- Computes a content hash for cache invalidation

Run this in your CI/CD pipeline so the index updates with every deploy.

### Step 2: Add edgekit to your site

```html
<div id="chat"></div>

<script type="module">
  import { createRuntime } from '@edgekit/core'
  import { webllm } from '@edgekit/model-webllm'
  import { localRAG } from '@edgekit/rag-local'
  import { webComponent } from '@edgekit/ui-component'

  const ui = webComponent({ theme: 'auto' })

  const runtime = createRuntime({
    model: webllm(),
    rag: localRAG({ indexUrl: '/content-index.json' }),
    ui,
    downloadPolicy: 'prompt',
    systemPrompt: `You are a documentation assistant. Answer questions
      using only the provided context. Cite the source page for each
      fact. If the context does not contain the answer, say so.`,
  })

  ui.mount(document.getElementById('chat'), runtime)
</script>
```

### Step 3: Deploy

Ship the content index JSON alongside your static site. The model weights are served from a CDN (no self-hosting required). Your deployment is:

- Your HTML/CSS/JS (as usual)
- `content-index.json` (built by the CLI)
- No new infrastructure, no API keys, no environment variables

## Example configuration

For a framework documentation site:

```typescript
const runtime = createRuntime({
  model: webllm({ modelId: 'Phi-4-mini-instruct-q4f16_1' }),
  rag: localRAG({
    indexUrl: '/content-index.json',
    topK: 5,
  }),
  downloadPolicy: 'prompt',
  systemPrompt: `You are the documentation assistant for FrameworkX.
    Answer questions using the provided documentation excerpts.
    Always cite the page title and section.
    If the docs don't cover the topic, say "This isn't covered
    in the current documentation" and suggest where to ask.`,
  guardrails: {
    maxInputTokens: 200,
    maxOutputTokens: 500,
    maxConversationTurns: 20,
  },
})
```

## What users experience

**First visit:** The user sees a chat widget. When they type a question, they are prompted to download the AI model (~2.5 GB). If they decline, they still get cited search results from the content index (retrieval-only mode).

**After model download:** Questions are answered in 1-3 seconds with streaming tokens. The model cites specific documentation pages. Users can ask follow-up questions. The experience feels like talking to someone who has read the entire documentation.

**Repeat visits:** The model is cached by the browser. Startup takes 2-5 seconds. No download prompt.

## ROI

The value proposition is straightforward:

| Metric | Before | After |
|---|---|---|
| Support tickets from "can't find in docs" | High | Reduced 40-60% |
| Time to answer for docs questions | Minutes (reading + searching) | Seconds |
| Cost per AI query | $0.003+ (cloud) | $0 |
| Privacy risk | User queries sent to third party | Zero (local inference) |
| Maintenance | API key management, usage monitoring | None |

For open-source projects especially, the zero-cost aspect is critical. Most OSS projects cannot afford cloud AI costs for their documentation. Edgekit makes AI search free to operate, regardless of traffic.

## Integrations

Edgekit works with any static site generator:

- **Astro**: Add the script to your layout component
- **Next.js**: Add to a client component
- **Docusaurus**: Add to a custom theme component
- **VitePress**: Add to a custom theme layout
- **Hugo / Jekyll**: Add the script tag to your template

The CLI processes standard markdown files. If your docs are in markdown (and most are), the indexing step works out of the box.
